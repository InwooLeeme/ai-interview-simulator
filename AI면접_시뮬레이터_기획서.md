# AI 역량면접 시뮬레이터 — 프로젝트 기획서 (v2)

> SK 스칼라(SKALA) AI 역량면접 환경을 재현하고, **지원자의 자기소개서·기술 내역서를 기반으로 맞춤 질문을 생성**하며, **면접 후 답변 피드백**까지 제공하는 웹 애플리케이션

---

## 1. 프로젝트 개요

### 1.1 배경
SK 스칼라 부트캠프 지원 시 진행되는 AI 역량면접은, 3명의 AI 면접관 앞에서 문항별로 짧은 고민시간과 정해진 답변시간 안에 응답해야 하는 독특한 환경이다. 실제 사람을 마주하는 면접과 달리 **환경 자체의 낯섦**(자동으로 흐르는 타이머, 버튼 조작, 카메라/마이크 응시 등)이 큰 부담 요소로 작용한다. 또한 실제 면접관이 제출 서류를 읽고 그에 맞춰 질문하듯, 시뮬레이터도 **지원자의 문서를 반영한 질문**을 던져야 연습 가치가 높다.

### 1.2 목표
- 실제 면접과 **동일한 흐름·압박감**을 재현해 "환경에 대한 적응"을 해결
- 지원자의 **자기소개서·기술 내역서를 기반으로 맞춤 질문**을 6~8문항 생성
- 답변을 **녹음·전사(STT)** 하여 스스로 복기
- 면접 종료 후 **답변별 피드백**(어떻게 답하면 좋았을지)을 제공

### 1.3 타깃 사용자
- SK 스칼라 등 AI 역량면접을 앞둔 취업/부트캠프 준비생 (본인 포함)
- 1인 사용 기준으로 시작 → 추후 공유 가능한 형태로 확장 가능

### 1.4 핵심 차별점
이 프로젝트는 단순 "면접 환경 시뮬레이터"를 넘어 **"내 서류를 읽고 묻고, 내 답변을 피드백하는 개인 맞춤 모의면접 도구"** 다. 따라서 가치의 두 축은 (A) 타이머·전환·음성 흐름의 **재현 정확도**와 (B) 문서 기반 질문 생성 및 피드백의 **개인화 품질**이다.

## 2. 핵심 기능 명세

실제 절차 + 신규 요구사항을 통합해 분해한다.

| 단계 | 기능 | 설명 |
|------|------|------|
| 0 | **문서 입력** (신규) | 면접 시작 전, 자기소개서·기술 내역서를 입력하는 화면 |
| 0.5 | **질문 생성** (신규) | 입력 문서를 LLM에 전달해 면접관별 6~8문항 생성 |
| 1 | 시험장 입장 (대기실) | 카메라·마이크 권한 요청, 환경 점검, 시작 안내 |
| 2 | 3명의 AI 면접관 표시 | 진행자 / 기술 전문 / 인성 전문 — 동시 배치, 말하는 면접관 하이라이트 |
| 3 | 문항 제시 | 담당 면접관이 음성(TTS)으로 질문을 읽어줌 |
| 4 | 고민시간 (10초) | 카운트다운, `답변하기` 버튼 활성화 |
| 5 | 답변시간 (3분) | 녹음 시작, 180초 카운트다운, `답변 완료` 버튼 |
| 6 | 자동 전환 | 고민 10초 경과 시 자동 답변 진입 / 답변 3분 경과 시 자동 종료 |
| 7 | 다음 문항·면접관 전환 | 시나리오대로 진행 |
| 8 | 면접 종료 | 진행자 마무리 멘트 |
| 9 | **피드백** (신규) | 답변별 전사 + LLM 피드백(개선 포인트, 모범 답변 방향) 제공 |

### 2.1 타이머·버튼 로직 (핵심 메커니즘)
- **고민시간(THINKING)**: 10초 카운트다운. `답변하기` 버튼 즉시 활성화 → 사용자가 일찍 누르면 곧바로 답변 단계. 누르지 않고 10초가 끝나면 **버튼이 자동 트리거되어** 답변 단계로 자동 진입.
- **답변시간(ANSWERING)**: 진입과 동시에 녹음 시작 + 180초 카운트다운. `답변 완료` 버튼으로 조기 종료, 안 누르면 180초 후 자동 종료. 종료 시 녹음 확정 → 다음 문항 전환.

이 흐름은 **유한 상태 기계(FSM)** 로 모델링한다 (7절).

## 3. 신규 요구사항 상세 설계

### 3.1 문서 입력 (요구사항 1·2)
면접 시작 전 **문서 입력 화면**을 둔다.

- **입력 항목**
  - 자기소개서 (textarea, 글자 수 표시)
  - 기술 내역서 / 경력·기술 스택 (textarea 또는 항목형 입력)
  - (선택) 파일 업로드: `.txt`, `.md`, `.pdf`, `.docx` → 텍스트 추출 후 textarea에 채움
- **검증**: 최소 글자 수 안내(예: 자소서 200자 이상 권장), 비어 있으면 일반 질문으로 폴백
- **저장**: 세션 동안 메모리/로컬에 보관. 민감 정보이므로 **서버 영구 저장은 기본 비활성**, 키 노출 방지 위해 LLM 호출은 서버 프록시 경유

### 3.2 문서 기반 질문 생성 (요구사항 1·3)
입력된 문서를 LLM에 전달해 **면접관 3인 × 역할에 맞는 질문 6~8개**를 생성한다.

- **문항 수 동적 조절(요구사항 3)**: 기본값 7문항, 사용자가 슬라이더/셀렉트로 6~8 사이 조절. 생성 프롬프트에 목표 개수를 변수로 전달하고, 결과 개수를 검증해 부족하면 재요청·초과하면 절단.
- **면접관별 배분 예시(7문항 기준)**: 진행자 1(자기소개/지원동기) + 기술 3 + 인성 3. 6문항이면 기술 2/인성 3, 8문항이면 기술 3/인성 4 식으로 규칙화.
- **개인화 근거 부착**: 각 질문에 "이 질문이 어느 문서 근거에서 나왔는지"(`sourceHint`)를 함께 생성하면, 사용자가 왜 이 질문을 받았는지 복기 시 이해하기 쉽다.

### 3.3 면접 후 피드백 (요구사항 4)
면접 종료 후 **각 답변에 대한 피드백**을 제공한다.

- **입력**: 질문 + 해당 답변의 STT 전사본 + (있다면) 원본 문서 맥락
- **출력(질문별)**:
  - 잘한 점 (강점)
  - 개선 포인트 (구체적, 실행 가능하게)
  - 답변 구조 제안 (예: STAR 기법 — 상황/과제/행동/결과)
  - "이렇게 답하면 좋았을" 모범 답변 방향(예시 문장)
- **종합 피드백**: 전반적 인상, 시간 활용(답변 길이), 반복되는 습관, 우선 개선 1~3가지
- **주의**: 점수화보다 **개선 방향 제시** 중심. 전사 오류 가능성을 감안해 "전사 기준 피드백"임을 명시.

## 4. 권장 기술 스택

학습 프로젝트인 점을 고려해 "처음엔 무료/브라우저 내장 → 이후 유료 API로 품질 업그레이드" 2단계 전략.

### 4.1 프론트엔드
| 영역 | 권장 | 이유 |
|------|------|------|
| 프레임워크 | **Next.js (React)** | API Route로 API 키를 서버에 은닉. 문서 입력→질문 생성→피드백 모두 서버 경유가 필요하므로 풀스택 프레임워크가 유리 |
| 상태/흐름 | **XState** (또는 `useReducer` FSM) | 단계 전이 + 지연 전이(타이머)에 최적 |
| 스타일 | Tailwind CSS | 빠른 UI |
| 녹음 | **MediaRecorder API** + `getUserMedia` | 브라우저 내장 |
| 문서 파싱(선택) | 클라이언트: `pdf.js` / 서버: `pdf-parse`, `mammoth`(docx) | 업로드 파일 텍스트 추출 |

### 4.2 음성 (TTS / STT)
| 기능 | MVP (무료) | 품질 업그레이드 (유료) |
|------|-----------|----------------------|
| TTS (면접관) | 브라우저 `SpeechSynthesis` (ko-KR) | 네이버 클로바 보이스 / ElevenLabs 다국어 (한국어 자연스러움 우수, OpenAI TTS는 영어 최적화라 한국어엔 비권장) |
| STT (답변 전사) | 브라우저 `Web Speech API` (ko-KR) | Soniox(한국어 실시간 강점) / 클로바 / OpenAI `gpt-4o-transcribe` |

### 4.3 LLM (질문 생성 · 피드백) — 이제 핵심 기능
| 기능 | 권장 |
|------|------|
| 질문 생성 | Claude / OpenAI / Gemini — 면접관 페르소나별 프롬프트 + 문서 컨텍스트 + 목표 문항 수 |
| 답변 피드백 | 동일 LLM에 질문+전사+루브릭 전달 |
| 출력 형식 | **JSON 구조화 출력**(아래 6절) — UI 매핑·검증 용이 |

### 4.4 백엔드 / 인프라
- **Next.js API Routes** (가장 간단) 또는 별도 FastAPI/Express
- 역할: LLM·STT·TTS 호출 **프록시(키 은닉)**, 문서 파싱, 질문/피드백 생성, 세션 임시 보관
- 데이터: 초기 IndexedDB/메모리 → 기록 누적 필요 시 SQLite/Supabase. 문서는 민감정보이므로 기본 비저장 권장

## 5. 구현 방법론 (How to Build)

### 5.1 전체 데이터 흐름
```
[문서 입력 화면]
   사용자: 자소서 + 기술내역서 + 목표 문항 수(6~8) 입력
        │  (파일 업로드 시 텍스트 추출)
        ▼
[POST /api/generate-questions]  ← 서버(키 은닉)
   서버: 페르소나 프롬프트 + 문서 + 목표 개수 → LLM 호출
        │  JSON(질문 배열) 반환 → 개수/형식 검증
        ▼
[면접 진행 (FSM)]
   질문 제시(TTS) → 고민10s → 답변3min(녹음) → 전환 ...
   각 답변: MediaRecorder Blob → STT 전사 텍스트 저장
        ▼
[POST /api/feedback]  ← 서버
   서버: (질문 + 전사 + 문서맥락 + 루브릭) → LLM 호출
        │  JSON(질문별 피드백 + 종합) 반환
        ▼
[피드백 화면] 질문·내 답변·피드백 나란히 표시
```

### 5.2 질문 생성 구현 방법
1. **서버 라우트 분리**: 클라이언트는 절대 LLM 키를 갖지 않는다. `/api/generate-questions`가 유일한 호출 지점.
2. **프롬프트 구성(권장 골격)**
   - *System*: "너는 SK 스칼라 AI 역량면접의 면접관 3인(진행자/기술/인성)이다. 지원자의 자기소개서와 기술 내역서를 근거로, 각 면접관 역할에 맞는 질문을 생성한다. 반드시 지정된 JSON 형식으로만 응답한다."
   - *User(변수 주입)*: 자기소개서 본문, 기술 내역서 본문, **목표 문항 수 N(6~8)**, 면접관별 배분 규칙.
   - *제약*: 문서에 등장한 구체적 경험·기술을 1개 이상 인용해 질문을 개인화할 것. 추측성 사실 단정 금지. 한국어로.
3. **구조화 출력 강제**: "JSON 외 텍스트·마크다운·코드펜스 금지"를 명시하고, 서버에서 파싱 실패 시 1회 재시도. (가능하면 모델의 JSON/구조화 출력 모드 사용)
4. **검증 단계**: 반환된 배열 길이가 N인지, 각 항목에 필수 필드가 있는지 확인. 부족하면 보충 요청, 초과하면 우선순위대로 절단.
5. **폴백**: 문서가 비었거나 LLM 실패 시, 하드코딩된 일반 질문 은행에서 N개를 채워 면접이 끊기지 않게 한다.

### 5.3 피드백 구현 방법
1. **수집**: 면접 종료 시점에 `answers[]`(질문 + 전사 + 사용 시간)가 모여 있다.
2. **호출 전략**: 답변별 개별 호출(피드백 품질↑, 호출 수↑) vs 전체 일괄 호출(비용↓). MVP는 **일괄 1회 호출 + JSON 배열 반환** 권장.
3. **프롬프트 골격**
   - *System*: "너는 면접 코치다. 각 질문과 지원자 답변(음성 전사본)을 보고, 비난이 아닌 **개선 중심**의 구체적 피드백을 한국어로 제공한다. 전사 오류 가능성을 감안한다. JSON으로만 응답."
   - *User*: 질문·전사·(문서 맥락)·평가 루브릭(명료성/구체성/직무적합성/구조 STAR).
4. **출력 매핑**: JSON 배열을 그대로 피드백 화면 카드에 매핑.
5. **안전장치**: 점수 단정 대신 방향 제시. 전사가 비었으면(무응답) "답변이 감지되지 않음" 처리.

### 5.4 음성 동기화 방법
- TTS는 **한 번에 한 면접관**만 재생(큐 직렬화). `SpeechSynthesisUtterance.onend`(또는 API 스트림 종료)로 다음 상태 전이.
- STT는 답변 단계 동안만 활성화, `답변 완료`/타임아웃 시 종료하여 전사 텍스트 확정.

### 5.5 개발 순서(권장)
문서→질문 생성을 **먼저 콘솔/포스트맨으로 검증**한 뒤 UI에 붙인다. 즉 "프롬프트가 원하는 JSON을 안정적으로 내놓는가"를 UI보다 먼저 확정해야 뒷단이 흔들리지 않는다.

## 6. 데이터 모델

```jsonc
// 문서 입력
{
  "resume": "자기소개서 본문...",
  "techProfile": "기술 내역서 본문...",
  "questionCount": 7            // 6~8, 사용자 조절
}

// 생성된 질문 (LLM 출력 형식)
{
  "id": "q-001",
  "interviewer": "technical",        // facilitator | technical | personality
  "text": "이력서에 적으신 ○○ 프로젝트에서 가장 큰 기술적 난관은 무엇이었나요?",
  "sourceHint": "기술 내역서의 ○○ 프로젝트 경험",  // 개인화 근거
  "thinkSeconds": 10,
  "answerSeconds": 180
}

// 답변 기록
{
  "questionId": "q-001",
  "audioBlobRef": "...",
  "transcript": "STT 전사 결과...",
  "durationUsed": 142
}

// 피드백 (LLM 출력 형식)
{
  "questionId": "q-001",
  "strengths": ["..."],
  "improvements": ["..."],
  "structureTip": "STAR로 정리하면: 상황... 과제... 행동... 결과...",
  "modelAnswerDirection": "핵심 성과를 수치와 함께 먼저 제시하고..."
}
// + 종합 피드백 객체(전반 인상, 시간 활용, 우선 개선 1~3)
```

## 7. 상태 기계(FSM) 설계

```
DOC_INPUT (자소서·기술내역서·문항수 입력)
   │ 제출
   ▼
GENERATING (LLM 질문 생성 + 검증, 로딩)
   │ 성공(질문 N개 확보)
   ▼
IDLE (대기실: 카메라·마이크 권한)
   │ 시작
   ▼
INTRO (진행자 인사 TTS)
   ▼
QUESTION_PRESENT (담당 면접관 질문 TTS)
   ▼
THINKING (10초, '답변하기' 활성)   ──버튼/10초경과──┐
   ▼                                              │
ANSWERING (180초, 녹음, '답변 완료') ─버튼/180초경과─┤
   ▼                                              │
TRANSITION (녹음·전사 저장 → 다음 결정) ◄───────────┘
   │ 남은 문항 → QUESTION_PRESENT
   │ 문항 끝 ▼
OUTRO (진행자 마무리)
   ▼
FEEDBACK_LOADING (LLM 피드백 생성)
   ▼
REVIEW (질문·내 답변·피드백 표시) → 종료
```

XState 사용 시 `THINKING`에 `after: { 10000: 'ANSWERING' }`, `ANSWERING`에 `after: { 180000: 'TRANSITION' }`를 선언하고 버튼 이벤트(`ANSWER_START`, `ANSWER_END`)로 동일 전이를 트리거하면 수동/자동 경로가 자연스럽게 합쳐진다. `GENERATING`·`FEEDBACK_LOADING`은 비동기 호출 상태로 두고 성공/실패 전이를 분리한다.

## 10. 기술적 난관 & 대응

| 난관 | 대응 |
|------|------|
| 타이머 중복·누수 | 상태 기계로 통합, 산발적 `setTimeout` 금지 |
| LLM JSON 깨짐 | 구조화 출력 강제 + 파싱 실패 시 재시도 + 폴백 질문 |
| 질문 개수 안 맞음 | 목표 N 검증 후 보충/절단 |
| 문서 민감정보 | 서버 영구 저장 기본 비활성, 키는 서버에만 |
| TTS 끝 시점 동기화 | `onend`/스트림 종료 이벤트로 전이 |
| 면접관 음성 겹침 | TTS 재생 큐 직렬화 |
| STT 전사 오류 | 피드백에 "전사 기준" 명시, 무응답 처리 분기 |
| API 키 노출 | 프론트 금지, API Route 프록시 경유 |
| 한국어 음성 품질 | MVP 브라우저 내장 → 한국어 특화 서비스 교체 |

## 11. 한 줄 요약
**"① 타이머·전환·음성 흐름을 상태 기계로 재현(Phase 1) → ② 자소서·기술내역서 기반 6~8문항 맞춤 생성(Phase 2) → ③ 답변 전사 후 개선 중심 피드백(Phase 3)."** 질문 생성과 피드백은 모두 서버 라우트에서 **JSON 구조화 출력**으로 받아 검증·매핑하는 것이 구현의 핵심이다.
