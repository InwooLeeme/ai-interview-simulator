# AI 역량면접 시뮬레이터 — 프로젝트 기획서 (v3, 에이전틱 아키텍처)

> AI 역량면접 환경을 재현하고, 지원자의 자기소개서·기술 내역서를 기반으로 맞춤 질문을 생성하며, 면접 후 답변 피드백까지 제공하는 **멀티 에이전트 기반** 웹 애플리케이션.

---

## 1. 프로젝트 개요

### 1.1 배경
AI 역량면접은 3명의 AI 면접관 앞에서, 문항별 짧은 고민시간과 정해진 답변시간 안에 응답해야 하는 독특한 환경이다. 자동 타이머·버튼 조작·카메라 응시 등 **환경 자체의 낯섦**이 큰 부담 요소이며, 실제 면접관처럼 **제출 서류를 읽고 그에 맞춰 질문**해야 연습 가치가 높다.

### 1.2 목표
- 실제 면접과 동일한 흐름·압박감을 재현해 "환경 적응" 문제를 해결한다.
- 자기소개서·기술 내역서 기반 **맞춤 질문 6~8문항**을 생성한다.
- 답변을 녹음·전사(STT)하여 복기할 수 있게 한다.
- 면접 후 **답변별 피드백**을 제공한다.
- 위 지능형 작업(질문 생성·진행·평가·피드백)을 단일 LLM 호출이 아니라, **역할이 분리된 다중 에이전트가 협업하는 에이전틱 구조**로 구현한다.

### 1.3 단일 LLM → 에이전틱으로 바꾸는 이유
v2는 "질문 생성 1회 호출 + 피드백 1회 호출"의 단순 파이프라인이었다. 그러나 이 도메인에는 본질적으로 **역할이 다른 3명의 면접관 + 진행 오케스트레이터 + 평가자**가 존재한다. 각 역할을 독립 에이전트로 분리하면:

- 면접관별 페르소나·관점이 한 덩어리 프롬프트에 섞이지 않고 **모듈화**된다.
- 진행 흐름(누가 언제 말하는가)을 **오케스트레이터 에이전트**가 명시적으로 제어한다.
- 꼬리질문·답변 기반 적응처럼 **상태에 따라 분기·반복하는 행동**을 자연스럽게 표현한다.
- 각 에이전트를 개별적으로 교체·개선·디버깅할 수 있다.

### 1.4 핵심 차별점
**"내 서류를 읽고, 역할이 다른 면접관 에이전트들이 협업해 묻고, 내 답변을 평가 에이전트가 피드백하는 개인 맞춤 멀티 에이전트 모의면접 도구."**

## 2. 핵심 기능 명세

| 단계 | 기능 | 설명 |
|------|------|------|
| 0 | 문서 입력 | 면접 전 자기소개서·기술 내역서·목표 문항 수 입력 |
| 0.5 | 질문 생성 (에이전트 협업) | 문서를 바탕으로 면접관 에이전트들이 역할별 6~8문항 생성 |
| 1 | 시험장 입장 (대기실) | 카메라·마이크 권한 요청, 환경 점검 |
| 2 | 3 면접관 표시 | 진행자/기술/인성 — 말하는 에이전트 하이라이트 |
| 3 | 문항 제시 | 담당 면접관 에이전트의 질문을 TTS로 출력 |
| 4 | 고민시간 (10초) | 카운트다운, `답변하기` 활성화 |
| 5 | 답변시간 (3분) | 녹음 + 180초 카운트다운, `답변 완료` |
| 6 | 자동 전환 | 고민 10초 / 답변 3분 경과 시 자동 진행 |
| 7 | 다음 문항·면접관 전환 | **오케스트레이터 에이전트**가 순서 제어 |
| 8 | 면접 종료 | 진행자 마무리 멘트 |
| 9 | 피드백 (평가 에이전트) | 답변별 전사 → 강점/개선/구조/모범방향 + 종합 |

### 2.1 타이머·버튼 로직 (핵심 메커니즘)
- **고민시간 (10초)**: `답변하기` 즉시 활성화 → 일찍 누르면 바로 답변 단계. 안 누르고 10초가 끝나면 **자동 트리거**되어 답변 단계로 진입.
- **답변시간 (3분)**: 진입과 동시에 녹음 시작 + 180초 카운트다운. `답변 완료` 또는 180초 경과 시 종료 → 녹음 확정 → 다음 문항 전환.

클라이언트 측 흐름은 **XState FSM**으로(§8), 서버 측 에이전트 협업은 **LangGraph 상태 그래프**로(§4) 모델링하며, 둘은 개념적으로 대응된다.

## 3. 신규 요구사항 상세 설계

### 3.1 문서 입력
- **입력**: 자기소개서(textarea), 기술 내역서(textarea/항목형), (선택) 파일 업로드(`.txt`/`.md`/`.pdf`/`.docx` → 텍스트 추출).
- **검증**: 최소 글자 수 안내, 비어 있으면 일반 질문으로 폴백.
- **저장**: 세션 한정 보관. 민감정보이므로 서버 영구 저장은 기본 비활성, 에이전트 호출은 서버 경유.

### 3.2 문서 기반 질문 생성 (에이전트 협업)
- **문항 수 동적 조절**: 기본 7, 사용자가 6~8 조절. 오케스트레이터가 목표 개수를 각 면접관 에이전트에 배분.
- **배분 규칙**: 진행자 1 + 기술 + 인성. (7 → 1/3/3, 6 → 1/2/3, 8 → 1/3/4)
- **개인화 근거(`sourceHint`)**: 각 질문에 문서 근거를 첨부.

### 3.3 면접 후 피드백 (평가 에이전트)
- **입력**: 질문 + 답변 전사본 + 문서 맥락.
- **출력(질문별)**: 강점 / 개선 포인트 / 구조 제안(STAR) / 모범 답변 방향.
- **종합**: 전반 인상, 시간 활용, 반복 습관, 우선 개선 1~3.
- **주의**: 점수화보다 개선 방향 중심, "전사 기준" 피드백임을 명시.

## 4. 에이전트 아키텍처

### 4.1 프레임워크 선택: 왜 LangGraph인가
요구사항은 LangChain / LangGraph / AutoGen 중 하나의 활용이다.

| 프레임워크 | 성격 | 적합성 |
|-----------|------|--------|
| **LangGraph** | 워크플로를 **상태 그래프(노드·엣지)** 로 모델링. 상태 지속·조건 분기·휴먼인더루프에 강함 | ★★★ 면접은 "정해진 단계 + 상태 + 사람이 루프에 답변 투입" 구조라 정확히 부합 |
| AutoGen | 에이전트 간 **자유 대화** 중심, 멀티에이전트 추론 품질 우수 | ★★ 면접관끼리 자유 토론이 아니라 순서 통제가 필요 → 과함 |
| LangChain(코어) | 모델·도구 통합과 빠른 구축에 강함 | ★ 단독으론 상태 제어가 약함. 단, LangGraph의 기반이라 함께 사용 |

**결정: LangGraph를 메인 오케스트레이션 프레임워크로 채택**한다(LangChain 생태계 위에서 동작하므로 모델·도구 추상화도 함께 사용). 근거:

1. 면접 진행은 **명확한 단계와 상태**(어느 문항인지, 답변이 쌓였는지)를 들고 흐른다 → 그래프 + 상태 모델과 일치.
2. 면접자는 그래프 중간에 답변을 투입하는 **휴먼인더루프** 노드다 → LangGraph의 강점.
3. 프론트엔드 XState와 백엔드 LangGraph가 **개념적으로 대응**되어 설계 일관성이 높다.
4. 꼬리질문·재질문 같은 **조건 분기/반복**을 명시적으로 표현할 수 있다.

> Microsoft 생태계(.NET 등)이거나 면접관끼리의 자유 토론 자체를 실험하려면 AutoGen도 대안이다. 다만 본 프로젝트의 "통제된 순서 진행" 요구에는 LangGraph가 더 맞는다.

### 4.2 에이전트 구성 (역할 분리)
| 에이전트 | 역할 | 주요 입력 | 주요 출력 |
|---------|------|----------|----------|
| **Orchestrator** (진행 오케스트레이터) | 전체 흐름 제어: 질문 수 배분, 면접관 호출 순서, 종료 판단 | 세션 상태, 목표 문항 수 | 다음 행동(어느 면접관 호출) |
| **Facilitator** (진행자) | 인사·지원동기·자기소개 질문, 마무리 멘트 | 자소서 요약 | 질문/멘트 |
| **Technical** (기술 면접관) | 기술 내역서 기반 직무·기술 질문 | 기술 내역서 | 기술 질문 |
| **Personality** (인성 면접관) | 협업·가치관·태도 질문 | 자소서 | 인성 질문 |
| **Evaluator** (평가/피드백) | 답변 전사 평가, 개선 중심 피드백 | 질문+전사+문서 | 질문별·종합 피드백 |
| **Probe** (꼬리질문, 선택) | 답변 내용 기반 후속 질문 생성 | 직전 답변 전사 | 꼬리질문 |

각 에이전트는 **독립된 시스템 프롬프트(페르소나) + 전용 출력 스키마**를 가지며, 공유 상태(State)를 통해 통신한다.

### 4.3 LangGraph 상태(State) 설계
그래프 전체가 공유하는 상태 객체(예시):

```python
class InterviewState(TypedDict):
    resume: str             # 자기소개서
    tech_profile: str       # 기술 내역서
    target_count: int       # 6~8
    questions: list[dict]   # 생성된 질문들
    current_index: int      # 진행 중인 문항
    answers: list[dict]     # {question_id, transcript, duration}
    feedback: list[dict]    # 평가 결과
    phase: str              # generating | interviewing | feedback
```

### 4.4 그래프 구조 (노드 & 엣지)
```
[START]
   │
   ▼
(generate_questions)   Orchestrator가 면접관 에이전트들을 호출,
   │                   역할별 질문 생성 후 questions에 누적·검증
   ▼
(await_answer) ◄───────────────┐   휴먼인더루프: 면접자 답변(전사) 투입 대기
   │                           │
   ▼                           │
(route_next) ── 조건 엣지 ──────┤
   │  남은 질문 있음 → await_answer (current_index+1)
   │  (선택) 꼬리질문 조건 → probe → await_answer
   │  질문 소진 ▼
(evaluate)   Evaluator가 answers 전체를 평가해 feedback 생성
   │
   ▼
[END]
```

- **generate_questions**: 내부에서 Facilitator/Technical/Personality 에이전트를 순차(또는 병렬) 호출. Orchestrator가 `target_count`를 역할별로 배분하고, 합쳐서 개수·형식을 검증.
- **await_answer**: 그래프를 **interrupt**시켜 클라이언트의 답변 전사가 도착할 때까지 대기(휴먼인더루프).
- **route_next**: 상태(`current_index`, 꼬리질문 여부)에 따라 분기하는 조건 엣지.
- **evaluate**: Evaluator가 질문별·종합 피드백 산출.

### 4.5 상태 지속성(Persistence)
LangGraph **체크포인터**(메모리/SQLite 등)로 세션 상태를 저장하면, `await_answer`에서 interrupt된 뒤 답변이 도착했을 때 같은 그래프를 이어서 재개할 수 있다. "중간에 사람을 기다렸다 재개"하는 면접 패턴에 필수적이다.

## 5. 권장 기술 스택

학습 프로젝트인 점을 고려해 "무료/내장 → 유료 품질 업그레이드"의 2단계 전략을 따른다.

### 5.1 프론트엔드
| 영역 | 권장 | 이유 |
|------|------|------|
| 프레임워크 | **Next.js (React)** | 클라이언트 UI + 에이전트 서버 호출 게이트웨이 |
| 상태/흐름 | **XState** | 타이머·단계 전이(클라이언트). 서버 LangGraph와 개념 대응 |
| 스타일 | Tailwind CSS | 빠른 UI |
| 녹음 | MediaRecorder + getUserMedia | 브라우저 내장 |

### 5.2 에이전트 백엔드 (신규 핵심)
| 영역 | 권장 | 이유 |
|------|------|------|
| 에이전트 오케스트레이션 | **LangGraph** (+ LangChain 코어) | §4 근거. Python 생태계가 가장 성숙 |
| 에이전트 런타임 서버 | **FastAPI** (Python) | LangGraph가 Python 중심 → 별도 Python 서버. Next.js가 호출 |
| LLM 제공자 | Claude / OpenAI / Gemini 중 택1 | LangChain 추상화로 교체 용이 (model-agnostic) |
| 관측/디버깅 | **LangSmith** (선택) | 노드별 상태 추적·재현 |
| 상태 지속 | LangGraph 체크포인터 (메모리 → SQLite) | 휴먼인더루프 재개 |

> **아키텍처 형태**: Next.js(프론트) → FastAPI(LangGraph 에이전트 서버) → LLM. Next.js의 API Route는 얇은 프록시로 두고, 실제 에이전트 그래프는 Python 서버가 담당. (모두 JS로 가려면 LangGraph.js도 있으나 생태계 성숙도·예제는 Python이 우위.)

### 5.3 음성 (TTS / STT)
| 기능 | MVP (무료) | 업그레이드 (유료) |
|------|-----------|------------------|
| TTS (면접관) | 브라우저 SpeechSynthesis (ko-KR) | 네이버 클로바 보이스 / ElevenLabs (OpenAI TTS는 영어 최적화라 한국어엔 비권장) |
| STT (답변 전사) | 브라우저 Web Speech API (ko-KR) | Soniox / 클로바 / OpenAI gpt-4o-transcribe |

## 6. 구현 방법론

### 6.1 전체 데이터/제어 흐름
```
[문서 입력 화면]  자소서 + 기술내역서 + 문항수(6~8)
      │
      ▼  POST /api/interview/start  (Next API → FastAPI)
[LangGraph: generate_questions]
   Orchestrator → Facilitator/Technical/Personality 호출
   → questions 생성·검증 → await_answer에서 interrupt
      │ (questions 반환)
      ▼
[면접 진행: 클라이언트 XState FSM]
   질문 TTS → 고민 10s → 답변 3min(녹음) → STT 전사
      │
      ▼  POST /api/interview/answer  (전사 1건 전송)
[LangGraph 재개]  route_next 조건 분기
   남은 질문 → 다음 질문 반환(다시 interrupt)
   (선택) 꼬리질문 → probe 노드
   질문 소진 → evaluate 노드
      │
      ▼  evaluate 완료
[피드백 화면]  질문·내 답변·피드백(질문별 + 종합) 표시
```

### 6.2 에이전트별 구현 가이드
1. **공통**: 각 에이전트는 (시스템 프롬프트 = 페르소나) + (출력 스키마 = 구조화 JSON)로 정의. LangChain 구조화 출력/파서로 스키마를 강제하면 검증이 쉽다.
2. **Orchestrator**: `target_count`를 역할별로 배분하는 규칙 함수 + "다음에 누구를 호출할지" 결정. 단순 규칙(결정론적)으로 시작하고, 필요 시 LLM 기반 라우팅으로 고도화.
3. **면접관 3종**: 각자 문서의 관련 부분만 컨텍스트로 받게 한다(진행자=자소서 요약, 기술=기술내역서, 인성=자소서 가치관 부분). 각 질문에 `sourceHint` 포함을 지시.
4. **Evaluator**: `answers` 전체를 입력받아 일괄 평가(MVP). 비난 금지·개선 중심·전사 오류 감안·JSON 출력을 시스템 프롬프트에 명시.
5. **Probe(선택)**: `route_next`에서 "직전 답변이 충분히 구체적이지 않으면 1회 꼬리질문" 조건으로만 진입. **최대 횟수 가드**로 무한 루프 방지.

### 6.3 휴먼인더루프 구현 포인트
- `await_answer`에서 LangGraph **interrupt**로 그래프를 멈추고 현재 질문을 클라이언트에 반환.
- 클라이언트가 답변 전사를 보내면 **같은 `thread_id`(체크포인트)** 로 그래프를 resume → `route_next`가 다음을 결정.
- `thread_id` = 세션 ID로 관리. 체크포인터에 상태가 보존되므로 새 HTTP 요청에서도 이어진다.

### 6.4 검증·폴백·안전장치
- 질문 개수가 `target_count`와 다르면 보충/절단.
- 구조화 출력 파싱 실패 시 1회 재시도 → 그래도 실패면 폴백 질문 은행.
- Probe 최대 횟수 제한으로 종료 보장(그래프가 END에 수렴).
- 무응답(빈 전사)은 `evaluate`에서 "답변 감지 안 됨"으로 분기.

### 6.5 개발 순서 (권장)
1. **그래프 골격 먼저**: LLM 없이 더미 노드로 `generate → await → route → evaluate` 흐름과 interrupt/resume이 도는지 확인.
2. 면접관 에이전트 1종(기술)만 붙여 질문 생성·구조화 출력 안정화.
3. 나머지 에이전트 + Orchestrator 배분 규칙 추가.
4. 클라이언트(XState) 연결 → 타이머·녹음·전사.
5. Evaluator(피드백) 연결.
6. (선택) Probe·LangSmith 관측·유료 음성 교체.

> **핵심 원칙**: 에이전트를 늘리기 전에 **그래프의 상태 전이와 휴먼인더루프 재개가 먼저 안정**돼야 한다. 여기서 무너지면 에이전트를 아무리 잘 만들어도 진행이 꼬인다.

## 8. 클라이언트 상태 기계(FSM) — LangGraph와의 대응

```
DOC_INPUT → GENERATING (= 서버 generate_questions) → IDLE (대기실)
   → INTRO → QUESTION_PRESENT (TTS)
   → THINKING (10s, 자동/수동) → ANSWERING (180s, 녹음, 자동/수동)
   → TRANSITION (전사 전송 = 서버 await_answer resume → route_next)
        ├ 남은 질문 → QUESTION_PRESENT
        └ 소진 → OUTRO → FEEDBACK_LOADING (= 서버 evaluate) → REVIEW → END
```

클라이언트 FSM의 각 전이는 서버 LangGraph 노드와 거의 1:1로 대응한다. 클라이언트는 "타이머·녹음·표시", 서버 그래프는 "질문 생성·라우팅·평가"를 맡는 역할 분담이다.
